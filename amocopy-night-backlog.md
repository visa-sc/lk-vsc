# Ночной беклог crm.voyotravel.ru (согласован с Андреем 06.07.2026)

Цель: за несколько ночей превратить слепок amoCRM на crm.voyotravel.ru из «только чтение»
в работающую свою CRM (данные уже есть: 290k сделок, 245k контактов, 1.7M задач, 1.7M примечаний).

## ЖЕЛЕЗНЫЕ ПРАВИЛА НОЧНОЙ РАБОТЫ (нарушать нельзя)
0. ТОЛЬКО копия amoCRM 1-в-1 (решение Андрея 13.07): никакого нового функционала
   (лояльность, свои идеи, улучшения UX сверх амо) — всё новое позже отдельным этапом.
   Мерило любого решения: «так было в амо?» Если нет — не делаем.
1. Трогать ТОЛЬКО crm.voyotravel.ru: файлы public/amocrm_copy.html, amocopy*.js, tools/amoCopy*,
   отдельный сервис crm-edit (порт 3001, pm2-процесс amocopy-svc) и nginx-конфиг voyotravel-crm
   (только location-блоки crm-домена, всегда через nginx -t + reload, не restart).
2. НЕ трогать: server.js, клиентский ЛК, /admin, /vsc, /calc, vsc./dev., pm2-процесс «voyo»,
   основной nginx, живую amoCRM (в неё вообще не ходить, кроме согласованного синка на 1 rps).
3. После каждого деплоя — health-check: crm. (страница+API) И контрольно все прод-страницы
   (/, /cabinet, /vsc, /admin — они должны быть 200 и НЕ перезапускаться). Любая деградация →
   немедленный откат из git, разбор, только потом дальше.
4. Каждый завершённый блок — commit+push (мелкими коммитами). Утром — отчёт в чат.
5. Диск: следить на каждом шаге; при <1.5 ГБ свободного — остановить тяжёлые операции.
6. Все ночные изменения данных — в НОВОЙ БД (.amocopy-db/), слепок .amocopy/ не модифицировать
   (он — эталон). Правки поверх слепка хранятся отдельно, с логом кто/когда/что.

## НОЧЬ 1 — фундамент: своя БД + изолированный сервис записи
- [ ] Отдельный сервис amocopy-svc (node, порт 3001, отдельный pm2-процесс):
      auth тем же кодом/сессиями (.amocopySessions.json), REST для чтения/записи.
- [ ] nginx: в конфиг voyotravel-crm добавить location /edit-api/ → 127.0.0.1:3001
      (nginx -t, systemctl reload nginx; основной сайт не трогается).
- [ ] Импорт слепка в SQLite (better-sqlite3, .amocopy-db/crm.db):
      pipelines, statuses, users, cf-описания; leads, contacts, companies, tasks
      (+cf-значения; примечания — если хватит диска, иначе остаются bucket-файлами).
      Индексы: leads(pipeline,status,updated), contacts(phone-цифры), tasks(entity,till,done).
- [ ] Страница: чтение канбана/карточек/списков переключить на новый API (fallback на старый).
- [ ] Смоук-тесты: канбан, карточка, поиск, список контактов; сравнение счётчиков с .amocopy.

## НОЧЬ 2 — редактирование (копия становится CRM)
- [ ] Drag-n-drop сделки между этапами в канбане (+запись в БД + лента "изменён этап").
- [ ] Редактирование в карточке сделки: название, бюджет, ответственный, значения полей.
- [ ] Создание: новая сделка (из шапки), новый контакт, привязка контакт↔сделка.
- [ ] Задачи: создать в карточке, выполнить с результатом, перенести срок.
- [ ] Примечания: добавить текстовое примечание в карточку.
- [ ] Журнал изменений (кто/когда/что) + вкладка «История» в карточке.

## НОЧЬ 3 — разделы amo и автоматизации v1
- [ ] Раздел «Задачи»: список по менеджерам, просроченные/сегодня/завтра, выполнение из списка.
- [ ] Движок автоматизаций v1 (по нашей описи + выгруженным digital_pipeline_*.json):
      при смене этапа → создать задачу по шаблону / сменить ответственного.
      Письма/SMS/боты — заглушки-логи «здесь будет отправка» (сервисы подключим отдельно).
- [ ] Конструктор-просмотр автоматизаций: человекочитаемая таблица триггеров по этапам.
- [ ] Рабочий стол: живые KPI из БД (заявки сегодня/вчера, по этапам, по менеджерам).

## НОЧЬ 4 — синхронизация и полировка
- [ ] Инкрементальный дозабор из живой amoCRM (updated_at-курсор, СТРОГО 1 rps, только GET):
      новые/изменённые сделки-контакты-задачи → БД; локальные правки не затираются (амо-версия
      помечается, конфликт показывается в карточке).
- [ ] Поиск по сделкам из шапки (название/id/бюджет) через SQLite FTS или LIKE-индекс.
- [ ] Пагинация «Ещё» в длинных этапах канбана, счётчики после правок, полировка UI по мелочи.
- [ ] Итоговый отчёт: что осталось до полного ухода с amoCRM (внешние каналы: телефония,
      Wazzup, SMS.RU, почта — по отдельному плану, не ночному).

## Не делаем ночами (по решению Андрея)
- SMS.RU, Wazzup, телефония, почтовые интеграции — трудоёмкие внешние подключения.
- Любые правки основного прода и живой amoCRM.

## Процедура запуска ночи
Андрей вечером пишет: «Запускай ночь N». Я работаю автономно до утра по этому файлу,
отмечаю чекбоксы, коммичу, утром — отчёт таблицей + что проверить глазами.
Диск: перед ночью 1 желательно расширение VPS (для SQLite с примечаниями); если не успеем —
ночь 1 идёт в компактном режиме (без примечаний в БД, они остаются в bucket-файлах слепка).
